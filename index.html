§ 00

The core framing

Modern systems are data-intensive, not compute-intensive. The bottleneck isn't CPU cycles — it's the amount, complexity, and change-rate of data.

Most interesting systems today are assembled from a small set of building blocks: databases, caches, search indexes, stream processors, batch processors, message queues. The engineering challenge is composing them reliably.

Kleppmann argues every data system must consciously design for three concerns. The rest of the book is organized around them — and so is this note.

The three pillars Kleppmann uses to organize the entire book. Every chapter that follows is a load-bearing detail of one of these.
§ 01

Reliability

"Continuing to work correctly, even when things go wrong."

Things that go wrong are faults; a system that anticipates faults and copes with them is fault-tolerant. A failure is when the system as a whole stops providing service.

The distinction matters: faults are local, failures are systemic. Good design prevents faults from cascading into failures.

A fault is one component deviating from spec. It only becomes a failure when it cascades.

Three classes of faults

Hardware faults

Disks die (~2–5 year MTTF), RAM corrupts, power supplies fail, networks drop. Traditionally handled via redundancy (RAID, dual power, etc.). As systems scale to thousands of machines, hardware fault rate per-fleet becomes high enough that software-level fault tolerance is required — you can't just keep adding hardware redundancy.

Software errors

Systematic bugs that correlate across nodes: a bad input that crashes every replica, a runaway process that eats all memory, a cascading failure where one slow service backs up another. Harder than hardware faults because they're not independent — redundancy doesn't help if the same bug runs on every node.

Human errors

Operators misconfigure things. Kleppmann cites a study showing configuration errors by operators were the leading cause of outages, while hardware faults played a role in only 10–25%. Mitigations: good abstractions, sandboxes/staging, thorough testing, fast rollback, good monitoring.

◆ For AI Systems
  • Model serving. A single corrupted weight file on one replica is a hardware fault. A bad model version pushed to all replicas is a software fault — and far worse. Canary deploys and shadow traffic are the answer.
  • Training runs. Long jobs on thousands of GPUs hit the "hardware faults are certain at scale" regime. Checkpointing is fault tolerance made concrete: tolerate node loss by resuming, not restarting.
  • Data pipelines. A silently-corrupt training record is a fault; it becomes a failure only if it poisons the model. Schema checks and data validation are parity bits for ML data.
  • Human error dominates LLM infra outages too. Wrong prompt template deployed, wrong model routed to a customer tier, feature flag flipped in prod instead of staging. Treat prompts and configs as code: reviewed, versioned, canaried.
§ 02

Scalability

"The ability to cope with increased load." Not a one-dimensional property — a specific question about how the system grows.

Describing load

You need load parameters before you can talk about scaling. Load parameters depend on the system: requests/sec, read/write ratio, cache hit rate, number of simultaneous active users.

Kleppmann's canonical example is the Twitter home timeline: posting a tweet is cheap, but delivering it to all followers (fan-out) is the load that dominates the design. The choice of load parameter determines the architecture.

Describing performance

Once load is defined, ask two questions: when load increases and resources stay constant, what happens to performance? When load increases, how much do you grow resources to keep performance constant?

The mean sits near p50 — it tells you nothing about the tail. SLOs should be defined in percentiles for this reason.

Latency vs response time

Latency is the duration a request waits to be handled. Response time is what the client observes — latency plus service time plus network delay. They are not the same.

Tail latency amplification

If a single user request fans out to 100 backend calls, the user's perceived response time is governed by the slowest of 100 — which approaches p99 of the backend. Your backend's p99 becomes the user's p50. This is why tail latency matters disproportionately in distributed systems.

An agent fanning out to 5 backend calls inherits the slowest one. With 100 calls, the user sees something like the backend's p99 as their p50.
◆ For AI Systems
  • Load parameter for LLM inference is not QPS. It's closer to tokens per second. A 50-token request and a 5,000-token request are wildly different load. Autoscaling on request count under-provisions long outputs and over-provisions short prompts.
  • Tail latency matters more for agents. An agent that makes 20 sequential LLM calls inherits p99 on each. Speculative decoding and request deduplication are tail interventions.
  • Training scales differently from inference. Training is throughput-bound (tokens/sec/GPU); inference is latency-bound (time-to-first-token). Same hardware, different load parameters, different architectures.
  • Vector DBs change architecture per order of magnitude. 1M vectors: brute force. 100M: HNSW or IVF. 10B: sharding plus hierarchical indexes.
§ 03

Maintainability

"Many people working on it over time must be able to do so productively." Most of a system's cost is paid after it's built.

Operability favors stability; evolvability favors change. Simplicity is the bridge that keeps both possible.

Operability

"Make it easy for operations teams to keep the system running smoothly." Good monitoring, clear runbooks, predictable behavior, sensible defaults, the ability to take individual nodes down without affecting the whole, and avoiding hidden dependencies on specific machines.

Simplicity

"Make it easy for new engineers to understand the system." Complexity slows down everyone who touches the system and increases the chance of bugs. Kleppmann names the antidote: abstraction — hiding implementation details behind a clean interface so you can reason locally.

Evolvability

"Make it easy to make changes in the future." Requirements change; the system must bend without breaking. Agile and TDD address evolvability at the code level; this chapter is about evolvability at the system architecture level — schema migrations, API versioning, data model changes.

◆ For AI Systems
  • Operability. Can you trace a bad response from a user back through the prompt template, retrieved docs, model version, and tool calls? If yes, your system is maintainable. If no, it's magic. LLM observability (Langfuse, Arize, etc.) is the AI analog of logs and metrics.
  • Simplicity. RAG systems sprawl: re-rankers on re-rankers, five chained LLM calls, bespoke caching at each layer. Each was added for a real reason; the sum is unmaintainable. Simplicity is an active fight.
  • Evolvability. The model under your system changes every 3–6 months. A maintainable AI system treats the model as a swappable component with a clear interface — not as load-bearing application logic. Otherwise every model upgrade is a rewrite.
§ 04

Synthesis

These three concerns are deliberately non-technical — they're not about Raft or LSM-trees. They're the lens you bring to every technical chapter that follows. When you read about replication (Ch. 5), you're reading about reliability and scalability tradeoffs. When you read about data models (Ch. 2), you're reading about maintainability and evolvability.

Chapter 1 is the scaffolding; the rest of the book hangs on it. For AI systems specifically, the mapping is clean:

Reliability
What happens when a model returns garbage, a GPU dies mid-batch, or a retrieval call times out?
Scalability
What's the actual load parameter — QPS, tokens/sec, concurrent users, context length?
Maintainability
Can a new engineer trace a prod issue in under an hour? Can we swap model versions without a rewrite?
§ 05

Key terms

Fault
One component deviates from its specified behavior.
Failure
The system as a whole stops providing the required service to the user.
Fault-tolerant
A system designed to anticipate faults and cope without failing.
Load parameter
The variable that describes how hard the system is working — application-specific (QPS, fan-out, hit rate, tokens/sec).
Latency
Time a request spends waiting to be handled.
Response time
What the client observes — latency plus service time plus network delay.
Percentile
The value below which a given fraction of observations fall. p99 = 99% of requests are faster than this number.
Tail latency amplification
When a request fans out to many backend calls, perceived latency approaches the backend's high percentiles.
Shared-nothing
Horizontally scaled architecture with no shared state between nodes.
Elastic
System that automatically adjusts resources to current load.
