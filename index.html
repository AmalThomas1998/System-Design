§ 0.1 — Chapter Overview Why foundations matter.

Every modern AI system is a distributed system first. A training run on 512 GPUs is a distributed system. An LLM serving 100K users is a distributed system. A vector database holding a billion embeddings is a distributed system. Skip the foundations and you'll spend years memorizing patterns you don't actually understand.

This chapter teaches you the language. Not vocabulary — the mental models that let you reason about any system you encounter, even ones that don't exist yet. CAP and PACELC aren't trivia; they're the framework you'll use when an interviewer asks how to design a globally consistent model registry. Sharding and consistent hashing aren't algorithms to memorize; they're the tools you'll reach for when your vector DB needs to scale 10×. Kafka isn't a product; it's the embodiment of an idea — durable async log — that shows up everywhere from training data ingestion to inference logging.

Most engineers skip this chapter and head straight to LLM serving. They learn vLLM and continuous batching and quantization. Then in an interview, when asked "how would you ensure your model registry stays consistent across regions during a partition?" — they freeze. Because they never learned the language.

You won't.

What you'll be able to do after this chapter

Four concrete capabilities you'll have by Week 4 end:

  • Look at any AI system component and immediately identify its consistency model, partitioning strategy, replication choice, and failure modes — without thinking about it.
  • Design a sharded, replicated, fault-tolerant data plane on a whiteboard in 20 minutes, with explicit quorum math and partition-handling decisions.
  • Architect a streaming pipeline (Kafka or equivalent) for ingesting training data or logging inference at any scale, with exactly-once semantics where required and at-least-once where it's not.
  • Explain why an LLM with 4× the context window costs ~16× more to serve, and reason about every tradeoff in transformer architecture from a systems perspective.

How this connects to later chapters: Chapter II (Data Layer) assumes you can reason about consistency, partitioning, and streaming. Chapter III (Training) assumes you understand bandwidth, replication, and fault tolerance. Chapter IV (Serving) assumes you can do back-of-envelope math on transformer costs. Chapter V (Modern Apps) assumes you understand all of it. Foundations isn't prerequisite material — it's the spine of the entire 24-week curriculum.

Distributed Systems Core

CAP, PACELC, consistency models, replication strategies.

✦

Core Concepts

With Tradeoffs

CAP Theorem

In a distributed system, when a network partition happens (and it will), you must choose between Consistency (every read sees the latest write) and Availability (every request gets a response). You cannot have both during a partition. Partition tolerance is non-negotiable in any real distributed system because networks fail.

The misconception: people think CAP is a one-time choice — "we're a CP system." Reality: a single application has different requirements for different data. Your model registry might be CP. Your inference cache is AP. Same company, same engineers, different choices.

When to use what CP for anything where stale data is dangerous (model versions in production, billing, auth tokens). AP for anything where availability beats accuracy (cached embeddings, recommendation feeds, analytics dashboards).

PACELC Theorem

CAP only describes behavior during partitions — which are rare. PACELC adds: Else, when running normally, choose between Latency and Consistency. So every database is classified as PA/EL, PC/EC, PA/EC, or PC/EL.

This is the more useful model. DynamoDB is PA/EL — sacrifices consistency for availability during partitions, sacrifices consistency for latency during normal operation. Google Spanner is PC/EC — chooses consistency in both modes, paying with availability and latency. Most production AI systems silently make these tradeoffs without engineers realizing it.

When to use what PA/EL when latency is user-facing critical (inference caches, session stores). PC/EC when correctness is non-negotiable (financial transactions, model version tracking). PA/EC is rare; PC/EL is rarer still and usually a config mistake.

Consistency Models (the hierarchy)

Not a binary "strong vs eventual" — there's a full ladder. From strongest to weakest: strict serializable → linearizable → sequential → causal → eventual. Stronger models are easier to program against but slower and less available. Weaker models are faster but require careful handling of stale reads.

Most engineers default to "strong consistency" because it sounds safer. But strong consistency across regions costs you 100ms+ on every write. Causal consistency — "I see my own writes, and order is preserved within a session" — is often the sweet spot for AI systems.

When to use what Linearizable: leader election, model version flags. Sequential: distributed counters. Causal: chat message ordering, agent state. Eventual: cache propagation, embedding sync, metrics.

Replication Strategies

Three main flavors. Single-leader: one node accepts writes, replicates to followers (PostgreSQL primary/replica). Simple, but the leader is a SPOF and writes don't scale. Multi-leader: several leaders accept writes, must resolve conflicts (active-active across regions). Complex, but high availability. Leaderless: any node accepts writes, uses quorums to ensure consistency (Cassandra, DynamoDB).

Replication is also synchronous (writes confirmed only after replicas ack — durable but slow) or asynchronous (writes confirmed immediately — fast but can lose data on leader failure). Most production systems use semi-sync: one sync replica, the rest async.

When to use what Single-leader for OLTP and most data stores. Multi-leader for global, write-heavy systems where conflicts are rare. Leaderless for high-write throughput where partial reads/writes are OK (logs, telemetry, embeddings).

Real-World Case Study

Amazon DynamoDB

How Amazon DynamoDB became the canonical PA/EL system.

Problem

Amazon's shopping cart service (early 2000s) needed to be available always — even during regional failures. A failed "add to cart" cost real money. Traditional relational DBs went down during partitions and had high cross-region latency.

Architecture

Leaderless replication across regions. Data partitioned via consistent hashing across hundreds of nodes. Each item replicated to N=3 nodes. Writes acknowledged after W=2 acks; reads from R=1 (eventual) or R=2 (quorum, strong-ish).

Why these choices

Availability was business-critical (cart service drives revenue). Eventual consistency was acceptable because cart conflicts could be merged client-side ("you had A in cart on phone, B on laptop → show A+B"). Latency had to be single-digit milliseconds for a checkout flow.

Tradeoffs accepted

Eventual consistency → users could occasionally see stale carts. No multi-row transactions → checkout had to be designed around it. Conflict resolution complexity pushed to application code (vector clocks, then later last-write-wins).

What breaks at scale

Hot partitions: if one user has 10M items, they overload one node (mitigated by adaptive capacity). Cross-region replication lag during failures can cause "lost writes" perceived as bugs. Conflict resolution gets gnarly when application semantics aren't commutative.

OpenAI's model registry (architectural inference based on public posts).

Problem

Multiple regions serving GPT models. "Which model version is live in production?" must be globally consistent — serving the wrong version is catastrophic (billing errors, behavior drift, eval mismatches).

Architecture

Strongly consistent metadata store (likely Spanner-like or a Raft-based system) holding model version pointers. Model artifacts themselves replicated eventually to regional CDNs. Version pointer change is atomic and globally serialized.

Why these choices

Metadata is small and rarely changes — affordable to make CP. Model weights are huge but immutable — eventual replication is fine, you just pin to a hash. The split decouples "what's live" from "where the bytes are."

Tradeoffs accepted

Version updates can take seconds globally during the consensus round. If the metadata store partitions, you can't deploy a new version (acceptable — you're still serving the old one). Cost: PC/EC systems are expensive and operationally complex.

What breaks at scale

Consensus throughput limits how often you can roll out (not a problem at this layer). If artifact replication is slow, regions can fall behind and start failing requests routed to outdated weights — needs careful canary logic.

Failure Scenarios

Real Production Pain

Things that go wrong (and have, in famous postmortems).

Split Brain

Network partition cuts a leader-follower pair into two halves. Both halves elect leaders. Both accept writes. When the partition heals, you have divergent state. GitHub's October 2018 outage was exactly this — MySQL primaries on both sides accepted writes for 43 seconds, leaving 24 hours of data reconciliation.

Silent Data Loss via Last-Write-Wins

In leaderless systems, concurrent writes to the same key are resolved by timestamp. If clocks drift even slightly, the "wrong" write wins. Cassandra deployments have lost data when NTP sync failed and a node's clock jumped backward.

Read-Your-Writes Violations

User updates their profile, immediately reads it back, sees the old version. Happens when reads go to a follower replica that hasn't caught up. Looks like a bug but is actually correct behavior under eventual consistency. Common in social media: "I just posted, where is it?"

Quorum Misconfiguration

Engineer sets R=1, W=1, N=3 for "performance." R+W ≤ N means no consistency guarantee. Reads can return stale data even without partitions. This silently breaks correctness in production for months before anyone notices because it works fine 99% of the time.

Cascading Failure Under Partition

Strongly consistent systems (CP) refuse to serve when a quorum is lost. But upstream services that depend on them don't have circuit breakers — they retry endlessly, exhaust connection pools, and take down adjacent services. Google's 2020 OAuth outage cascaded this way.

Mitigations Use proven consensus (Raft/Paxos) for leader election to prevent split-brain. Sync clocks via NTP and prefer logical clocks (vector clocks, hybrid logical clocks) over wall-clock timestamps for ordering. Implement read-your-writes by routing a user's reads to the same replica or by reading from the leader for a short window after a write. Always check R+W>N when designing quorums. Wrap every CP call in a circuit breaker with bounded retries.

Interview Tips

FAANG + AI Labs

Common questions and how to answer them.

Design a globally distributed model registry that tracks which model version is live in each region.

Framework: (1) Clarify scale — how many models, how often updates, how many regions. (2) Identify the consistency boundary — version pointer must be linearizable, model artifacts can be eventual. (3) Pick technology: Spanner-style for metadata, S3 + CDN for artifacts. (4) Discuss failure: what if a region can't reach the metadata store? Serve old version, alert. (5) Acknowledge tradeoffs: deploys take seconds globally; that's fine.

Walk me through CAP. Then PACELC. Why is PACELC more useful?

Framework: Define both. Then make the case: partitions are rare (maybe 0.01% of time); latency-vs-consistency tradeoffs are constant. PACELC categorizes how a system behaves always, not just during partitions. Cite DynamoDB (PA/EL) vs Spanner (PC/EC) as canonical examples.

Your inference cache is returning stale embeddings 0.5% of the time. How do you debug?

Framework: Hypothesize: (1) Replication lag — check follower lag metrics. (2) Quorum misconfig — verify R+W>N. (3) Cache TTL race — invalidations not propagating. Then propose mitigation: route reads to leader for hot keys, increase R, or accept staleness with a read-repair scheme.

Strong answer vs weak answer.

Weak "I'd use a database. We need ACID, so probably PostgreSQL. We can shard it later if needed."
Strong "Two distinct data layers: metadata (model versions, deployment state) needs linearizability — Spanner or CockroachDB. Artifacts (model weights) are immutable and large — S3 with regional caching. The pointer flip is the consistency boundary."
Weak "We'll replicate to three nodes for high availability."
Strong "N=3 replicas, W=2 / R=2 quorum gives us strong consistency. We accept one node failure with no impact, two failures with read-only mode. Cross-region replication is async to keep p99 writes under 50ms."

Mistakes candidates make.

  • Treating "consistency" as binary — strong vs eventual — and ignoring causal, sequential, and read-your-writes.
  • Saying "CA system" out loud. CA systems don't exist in distributed contexts; partition tolerance is mandatory.
  • Over-engineering for partitions that happen 0.01% of the time while ignoring the latency cost paid 100% of the time.
  • Forgetting to ask about scale before picking a consistency model. CP is fine for 100 writes/sec, painful at 100K/sec.
  • Not mentioning quorum math (R+W>N) when discussing leaderless systems. It's the entire point.
  • Picking strong consistency "to be safe" without justifying the cost. Defaults should be defended, not assumed.

Scaling Primitives

Sharding, consistent hashing, load balancing.

✦

Core Concepts

With Tradeoffs

Sharding (Horizontal Partitioning)

When one machine can't hold your data, split it across many. Sharding is horizontal partitioning by key — every record lives on exactly one shard, determined by some function of the primary key. The choice of shard key is the single most important decision and the hardest to change later.

Strategies: range-based (e.g., user IDs A–F on shard 1, G–M on shard 2) — easy to reason about, terrible for hot ranges. Hash-based (hash(user_id) mod N) — uniform distribution, but range queries become scatter-gather. Directory-based (lookup table from key to shard) — flexible, but the directory itself becomes a SPOF.

When to use what Range-based for time-series and analytics where you query ranges. Hash-based for OLTP where you access single records. Directory-based for systems that need to migrate shards frequently (multi-tenant SaaS).

Consistent Hashing

The naive sharding scheme — hash(key) mod N — has a fatal flaw: if you add or remove a node, almost all keys move. Consistent hashing fixes this. Hash both keys and nodes onto a ring (0 to 2^32). Each key goes to the next node clockwise. Adding or removing a node only affects keys in one slice of the ring — typically 1/N of total keys.

Virtual nodes (vnodes): each physical node owns multiple ring positions. This smooths load distribution and makes rebalancing graceful. Cassandra, DynamoDB, and most modern caches (Memcached, Redis Cluster) use this pattern.

When to use what Use consistent hashing whenever your node count changes — caches, distributed key-value stores, sharded inference services. Don't bother for fixed-size clusters (e.g., a 3-node leader/follower setup).

Load Balancing

Spreading requests across servers. Algorithms: round-robin (simple, ignores load), least connections (better, picks the least busy), weighted (when servers have different capacities), consistent hashing (sticky routing — same user → same backend, useful for session affinity and cache locality).

Layer matters. L4 (TCP/UDP) load balancers are fast but dumb — they can't read HTTP. L7 (HTTP) load balancers can route by path, header, cookies — slower but smarter. Modern AI inference services typically use L7 with consistent hashing on user_id to keep KV cache warm on the same GPU.

When to use what Round-robin for stateless workers. Least-connections for variable-cost requests (some inference calls are 5× longer than others). Consistent hashing when there's caching state to preserve (LLM KV cache, session data).

Hot Partitions and Rebalancing

The dirty secret of sharding: real-world key distributions are never uniform. One celebrity user in your social app gets 1000× the traffic of a normal user. One LLM session is 100× longer than average. Hot partitions overload one shard while the rest sit idle.

Mitigations: adaptive sharding (DynamoDB does this — splits hot partitions automatically), request hedging (read from multiple replicas of a hot partition), application-level salting (append a random suffix to hot keys, then aggregate at read time).

When to use what Always profile your key distribution before sharding. If you have power-law traffic (most production systems do), plan for hot keys upfront — pick a sharding scheme that supports adaptive splits or design for salting.

Real-World Case Study

Discord Message Storage

Discord migrated trillions of messages from Cassandra to ScyllaDB — sharding lessons at scale.

Problem

Discord stores trillions of chat messages. They sharded by (channel_id, bucket) where bucket is a time window. As popular channels grew, single buckets exceeded shard capacity. "Hot partitions" caused timeouts and cascading failures.

Architecture

Composite shard key: (channel_id, time_bucket). Hash distributes channels across nodes; time bucket prevents any single channel from hot-spotting one shard for too long. Bucket size was tuned to match expected message volume.

Why these choices

Pure channel_id sharding caused massive hot partitions for huge servers (Midjourney, MrBeast). Adding time bucket spread one channel's load across multiple shards over time, capping the per-shard write rate.

Tradeoffs accepted

Range queries (load latest 50 messages) now need to scan multiple buckets. Tombstones from deleted messages stuck around longer than expected — a separate Rust service was needed just to manage them. Migration to ScyllaDB took years of engineering.

What breaks at scale

Even with bucketing, viral moments still create hot spots within a single bucket. Deletes pile up tombstones that slow reads. Cross-shard transactions (e.g., "delete a user's last 100 messages") become application-coordinated and harder to reason about.

Failure Scenarios

Real Production Pain

Sharding sins and how they bite.

Bad Shard Key (the original sin)

Picking a low-cardinality or skewed shard key (e.g., country code, where 80% of users are in 3 countries). Shards 1, 7, 12 melt while the rest sit at 5% utilization. Re-sharding is the most expensive operation in distributed systems — you'll write code for months to fix this.

Hot Partition Death Spiral

One celebrity post / viral channel / popular product suddenly receives 1M req/sec. The shard holding it overloads, latency spikes, retries hammer it, full-on outage. Twitter has had multiple outages from exactly this when celebrities tweet.

Resharding Nightmares

You used hash(user_id) mod N. You need to add capacity. Adding nodes shuffles 90% of data. Some teams have done this with multi-month migrations involving dual-writes, backfills, and praying nothing breaks. Consistent hashing prevents this; everyone learns this lesson once.

Cross-Shard Transactions

Once you shard, transactions that span keys on different shards become hard. Two-phase commit kills latency; sagas leak partial state on failure. Most teams just don't allow cross-shard transactions and bend the data model to avoid them.

Load Balancer Stickiness Becomes a Liability

You used consistent hashing on user_id to keep KV cache warm. One mega-user routes to one pod and overloads it. Pod dies. Now stickiness re-hashes a million users at once, cold-starting half your fleet. This pattern caused several LLM serving outages in 2024.

Mitigations Choose shard keys with high cardinality and uniform distribution; profile real traffic before deciding. Use composite keys (user_id, time_bucket) when one dimension is skewed. Use consistent hashing from day one; even if you start with 3 nodes, you'll thank yourself later. For hot keys, use request hedging or salting. For cross-shard transactions, prefer sagas or redesign the data model. For LB stickiness, fall back to least-connections for VIPs identified by request rate.

FAANG + AI Labs

Common questions and how to answer them.

Design a sharded vector database for 1 billion embeddings.

Framework: (1) Calculate: 1B × 768 dims × 4 bytes = ~3 TB. Per-node 256GB RAM → need ~12 shards. (2) Pick shard key — embedding ID via consistent hashing. (3) Discuss query: ANN search must hit all shards (scatter-gather), then merge top-K. (4) Replication: N=3 for fault tolerance. (5) Discuss hot partitions: namespace-based filtering can hit one shard hard — use salting or namespace-aware sharding.

Your LLM inference service uses consistent hashing on user_id for KV cache locality. One user's request rate is 10× normal. What now?

Framework: Identify the issue: hot pod. Short term: rate-limit that user. Medium term: detect "VIP" users and override the hashing — round-robin them across pods (lose KV cache for them, accept the cost). Long term: split sessions — same user, different conversations → different shard keys. Show you understand the cache vs balance tradeoff.

Why use virtual nodes instead of one position per physical node?

Framework: Three reasons: (1) Smoother load distribution with fewer physical nodes. (2) Heterogeneous nodes — give a 2× larger box 2× the vnodes. (3) Faster rebalancing — losing a node spreads its work across many neighbors instead of dumping it on one. Bonus: makes adding a node spread incoming load gradually.

Strong answer vs weak answer.

Weak "We'll shard by user_id. As we grow, we add more shards."
Strong "Consistent hashing on user_id with 256 vnodes per physical node. R=2, W=2, N=3 for replication. Whales (top 0.1% by traffic) detected via real-time metrics get round-robin'd across pods to avoid hot partitions, with replicated state for cache."

Mistakes candidates make.

  • Forgetting that sharding decisions are nearly permanent. "We can always re-shard" is a junior answer.
  • Not asking about traffic distribution. Power-law traffic dominates real systems and changes the design entirely.
  • Ignoring the cost of cross-shard queries. "Just join across shards" is hand-waving.
  • Choosing range-based sharding for OLTP without acknowledging hot range issues.
  • Not mentioning vnodes when discussing consistent hashing. It's the production-grade version.

Async & Streaming

Kafka, exactly-once semantics, backpressure.

✦

Core Concepts

With Tradeoffs

Why Async Matters

Synchronous systems are simple but brittle. Service A calls service B which calls service C — if any link is slow, the whole chain is slow. If C is down, A is down. Async decouples them: A writes to a queue, B reads when ready. Failures isolate, slow consumers don't block fast producers, and you can replay history.

For AI systems specifically: every inference call is logged async. Every training job pulls data async. Every fine-tuning trigger is an async event. Get good at thinking in events, not requests.

When to use what Sync for user-facing reads where the user is waiting. Async for everything that can be deferred — logging, analytics, training data ingestion, eval pipelines, model evaluation runs, A/B test result aggregation.

Kafka Architecture

Kafka is a distributed, durable, partitioned, replicated log. Producers append to topics; consumers read by offset. Each topic has multiple partitions for parallelism. Each partition is replicated across brokers for fault tolerance. Consumers track their own offset — Kafka doesn't push, consumers pull.

Key insight: Kafka isn't a queue. It's a log. Messages aren't deleted when consumed — they sit there for a retention period (hours, days, weeks). Multiple consumer groups can read the same topic independently. This is why Kafka powers everything: real-time analytics, ML training, audit logs, change data capture, all from the same data.

When to use what Kafka for high-throughput, durable, replayable streams (training pipelines, inference logs). RabbitMQ/SQS for traditional queues with task semantics. Redis Streams for low-latency, smaller-scale needs. Pulsar when you need both pub/sub and queue semantics.

Delivery Guarantees

Three flavors. At-most-once: fire-and-forget, may lose messages (cheapest, used for metrics where some loss is OK). At-least-once: retry until ack'd, may duplicate (most common — your consumer must be idempotent). Exactly-once: each message processed exactly once even with failures (expensive, requires transactional producer + consumer + idempotent processing).

Kafka's "exactly-once semantics" (EOS) works only within Kafka — produce and consume in the same transaction. Once data leaves Kafka (writes to a database, calls an API), you're back to at-least-once unless that downstream is also transactional.

When to use what At-most-once: telemetry, debug logs. At-least-once + idempotent consumer: nearly everything (default choice). Exactly-once: financial transactions, model training data integrity, inference billing.

Backpressure

What happens when a consumer can't keep up? Three options. Buffer (queue grows until OOM — bad). Drop (lose data — sometimes OK). Push back (signal upstream to slow down — best). Kafka uses pull-based consumers, which is itself a form of backpressure: consumers read as fast as they can, no faster.

For AI pipelines: a slow training data validator can backpressure ingestion. A slow embedding service can backpressure prompt logging. Without backpressure, the system silently degrades — queues grow unbounded, latency hides until OOM.

When to use what Pull-based (Kafka) handles backpressure naturally. Push-based (most legacy queues) needs explicit rate limits. For HTTP APIs, return 429 with Retry-After. For internal services, use bounded queues with explicit drop or block policies.

Real-World Case Study

Uber's Streaming Platform

Uber processes trillions of events per day on Kafka — every ride, location ping, fare calculation.

Problem

Real-time decisions at massive scale: matching riders to drivers, surge pricing, fraud detection, ETA prediction. All require fresh data from many services. Synchronous coupling between dozens of microservices was untenable.

Architecture

Kafka as the central nervous system. Every service publishes events (ride_started, location_update, payment_processed). Hundreds of consumer groups subscribe — each owning a different decision (matching, pricing, ML feature pipelines). Kafka clusters tiered by priority: critical, standard, batch.

Why these choices

Decoupling — services evolve independently. Replayability — when ML models change, replay last 7 days to backfill features. Multi-tenancy — same event consumed by real-time matching, batch analytics, and fraud detection without coupling. Tiering — payments can't share a cluster with debug logs.

Tradeoffs accepted

Operational complexity — running petabyte-scale Kafka requires a dedicated team. End-to-end latency adds tens of milliseconds vs sync. Eventually consistent across consumers — different services see slightly different worldviews at any instant.

What breaks at scale

Hot partitions (a viral city during a major event). Consumer lag during deploys can balloon and cause cascading delays. Schema evolution: a producer changes a field, a consumer breaks — Uber built schema registries and compatibility checks just for this.

⚠

Failure Scenarios

Real Production Pain

How streaming pipelines fail in subtle, awful ways.

Consumer Lag Death Spiral

Consumer slows down (GC pause, slow downstream). Lag grows. Topic retention is 24 hours. Lag exceeds retention. Messages now silently lost — Kafka deletes them as they "age out." You discover the loss days later when a downstream report has gaps.

Duplicate Processing Without Idempotency

Consumer reads message, calls payment API, crashes before committing offset. On restart, reprocesses message → double charge. Stripe-like systems have safeguards; everyone else's homegrown systems don't until they get burned.

Schema Breaking Changes

Producer adds a required field. Consumer with old schema crashes parsing. Topic backs up. Other services depending on that consumer go down. The original producer's PR review didn't include the dozen consumers downstream.

Cross-Region Replication Loops

Multi-region Kafka with mirror-maker. A misconfigured replication policy creates a feedback loop — Region A → Region B → Region A. One company filled their entire Kafka storage in 12 hours from this bug.

Slow Consumer Bringing Down Producer

Producer has bounded send buffer. Slow downstream causes backpressure to producer. Producer's threads block on send. Producer's HTTP service queues up. Producer's HTTP service starts returning 500s. The "queue" was supposed to decouple them — but only if backpressure is handled at every layer.

Mitigations Alert on consumer lag, not just throughput. Make every consumer idempotent — stamp messages with UUIDs and dedupe at the sink. Use schema registries (Confluent, Apicurio) with compatibility enforcement before deploys. Audit replication topology and document who reads what. Set explicit producer timeouts and have a "drop, don't block" fallback for non-critical paths.

Interview Tips

FAANG + AI Labs

Common questions and how to answer them.

Design an inference logging pipeline for an LLM service handling 100K requests/sec.

Framework: (1) Math: 100K req/sec × ~5KB/log = 500MB/sec, ~43TB/day. (2) Producer: write to Kafka with at-least-once + UUID. (3) Topic: 100+ partitions for parallelism, replication factor 3. (4) Consumers: ETL to S3/Parquet for training data, separate consumer for real-time anomaly detection. (5) Backpressure: bounded producer buffer, drop debug-level on overflow. (6) Schema: Avro with registry. Bonus: discuss tiered storage for cost.

When would you NOT use Kafka?

Framework: (a) Low scale — Kafka has operational overhead; for <10K msg/sec, SQS or Redis Streams are simpler. (b) Strict ordering across keys — Kafka guarantees order within partition only. (c) Strict task semantics with retries and DLQs — RabbitMQ is more natural. (d) Sub-millisecond latency — Kafka is "near real-time," not "real real-time."

Explain exactly-once semantics. How do you actually achieve it end-to-end?

Framework: Kafka EOS works within Kafka via transactional producer + read-process-write atomicity. End-to-end (Kafka → external system) requires either: (1) an idempotent sink keyed by message ID (most common, most practical), (2) two-phase commit (rare, fragile), or (3) the outbox pattern for DB writes. Strong answer admits exactly-once is mostly a marketing term and idempotency is what actually works.

Strong answer vs weak answer.

Weak "Use Kafka for the queue. The consumer reads messages and writes to the database."
Strong "Kafka with N partitions matched to consumer parallelism. At-least-once delivery, idempotent writes via upsert on message UUID. Schema registry to prevent breaking changes. Lag alerts at 30 minutes; auto-scale consumers on sustained lag."

Mistakes candidates make.

  • Saying "exactly-once" without acknowledging it's mostly impossible end-to-end.
  • Not mentioning idempotency. It's the answer to 80% of streaming problems.
  • Forgetting to account for retention and lag in capacity planning.
  • Treating Kafka as a queue instead of a log. Logs replay; queues don't.
  • Ignoring schema evolution. Real systems break here, not on throughput.

Core Concepts

From a Systems Lens

Transformers (the systems view)

You don't need to derive the math. You need to understand: a transformer is a stack of identical layers, each containing a self-attention block and a feed-forward block. Inputs flow in as tokens (integers from a vocabulary), get embedded into vectors, pass through the stack, come out as predictions for the next token.

The systems-relevant facts: parameters live in the layers (a 70B model has ~70B float weights to load into GPU memory). Compute is dominated by matrix multiplications — perfect for GPUs. Memory is dominated by activations and the KV cache during inference — this is what limits batch size and context length.

What this means for systems More layers + wider layers → more parameters → more GPU memory needed. Bigger context → quadratic memory cost in attention. Bigger batch → linear memory increase but better GPU utilization.

Attention Mechanism

Attention answers: "for each token, which other tokens should I pay attention to?" It computes a weighted sum of all other tokens' representations, where the weights come from query–key similarity. Self-attention is O(n²) in sequence length — this is the central scaling problem of LLMs.

Doubling context length = 4× memory and compute for attention. 4× context = 16× cost. This is why 1M-token context windows are expensive and why optimizations like FlashAttention, sliding window attention, and KV cache reuse are entire research areas.

What this means for systems Long contexts shift cost from feed-forward to attention. Inference cost ≈ linear in tokens generated (autoregressive decode is one token at a time). Prefill cost ≈ quadratic in prompt length. This split is why "disaggregated serving" (separate prefill and decode workers) exists.

Embeddings

An embedding is a dense vector representation of something — a token, a sentence, an image. Models learn to place semantically similar inputs near each other in vector space. Cosine similarity (or dot product) measures how "related" two embeddings are.

For systems: embeddings are typically 384–4096 dimensions, float32 (16 bytes/dim) or float16 (8 bytes/dim) or quantized to int8 (1 byte/dim). A billion 1024-dim float32 embeddings = 4TB. This is why vector databases, quantization (PQ, OPQ), and approximate search (HNSW, IVF) exist.

What this means for systems Embedding generation is GPU-bound but cheap (a few ms per text). Embedding storage is memory-bound and expensive at scale. Search is latency-bound — exact search is O(N), approximate search is O(log N) but trades recall for speed.

The Inference Cost Equation

For an LLM, total cost ≈ (model size × sequence length × tokens generated) per request. A 70B model serving a 4K-token prompt and generating 1K tokens needs to load 70B params, compute attention over 4K tokens, then autoregressively generate 1K — each generation step does a forward pass on the cached state.

The KV cache is what makes this practical: instead of recomputing attention for the entire context every step, cache the keys and values from previous tokens and only compute new ones. KV cache size = 2 × layers × heads × head_dim × seq_len × batch_size × bytes-per-elem. For Llama-70B at 4K context: ~10GB just for the cache, per request.

What this means for systems Model size sets the floor on GPU memory. KV cache size sets the ceiling on concurrent requests. Context length squeezes both. Quantization (FP8, INT4) reduces both floor and KV-cache cost — at some accuracy cost.

Real-World Case Study

OpenAI's GPT Serving

How OpenAI serves GPT models at scale (architectural inferences from public talks & papers).

Problem

Serving GPT-4-class models to millions of users, with widely varying request shapes — short chats, long documents, code generation, all on the same model. Latency must be perceived as conversational; cost must be sustainable.

Architecture

Tensor + pipeline parallelism across many GPUs (one model spread across 8–16 GPUs). Continuous batching (vLLM-style) — new requests join an in-flight batch instead of waiting. KV cache managed in pages (PagedAttention). Speculative decoding with a smaller draft model to reduce decode latency. Prefill and decode often disaggregated to different worker pools.

Why these choices

Model is too big for one GPU → parallelism is mandatory. Static batching wastes GPU when requests have different lengths → continuous batching keeps utilization high. Naive KV cache fragments memory → paged caches recover lost capacity. Decode is memory-bound (small compute per step but loads weights every time) → speculation amortizes the load across multiple tokens.

Tradeoffs accepted

Engineering complexity is enormous — vLLM, PagedAttention, speculative decoding are research-grade systems. Speculative decoding adds tail-latency variance. Disaggregation adds network hops. Cost per token still dominates the unit economics conversation.

What breaks at scale

Mega-prompts (100K+ tokens) blow up KV cache — must isolate to specific worker pools. Bursty traffic from a single key user — same hot-partition problem from Week 2. Long-tail of cold sessions — hard to keep useful KV cache without OOM. Quantization regressions creep in subtly across model versions.

Failure Scenarios

Where ML Systems Bleed

How transformer-based systems fail in production.

OOM from a Single Long Prompt

Average prompt is 500 tokens. One user sends 100K. KV cache for that one request balloons, GPU OOMs, entire pod crashes, in-flight requests for OTHER users die. Hard cap on prompt length and per-request memory budgets are non-negotiable.

Throughput Cliff at Long Context

Throughput is fine at 4K context, drops 10× at 32K. Why? Quadratic attention cost; you're now compute-bound where you were memory-bound. Capacity planning based on average context misses the long-tail entirely.

KV Cache Fragmentation

Without paged attention, allocating one big block per request leaves "holes" as requests finish. A new long request needs contiguous memory but only fragments are free. Throughput collapses even though "free memory" looks fine.

Quantization Drift

Team quantizes the model from fp16 to int8 to save memory. Eval suite shows 0.3% accuracy regression — "good enough!" Three weeks later, support tickets pile up: certain edge cases (math, code, non-English) regressed by 10–30%. Eval coverage was the gap, not the quantization.

Embedding Index Drift

You re-embedded all documents with a new model. Half your queries still use the old model. Queries and documents are now in different vector spaces. Search "works" but recall silently dropped 40%. Vector spaces aren't comparable across model versions.

Mitigations Hard prompt-length caps with explicit error responses. Per-request memory budgets and admission control. Use PagedAttention or equivalent KV cache management (vLLM, TensorRT-LLM). Build domain-specific eval suites that exercise edge cases before quantizing. Always re-embed query and document corpus together; tag embeddings with model version and refuse cross-version search.

Interview Tips

FAANG + AI Labs

Common questions and how to answer them.

Why does doubling context length cost more than 2× to serve?

Framework: Three reasons: (1) Attention is O(n²) — 2× context = 4× attention compute and memory. (2) KV cache scales linearly per request, so you fit fewer concurrent requests per GPU — utilization drops. (3) Prefill takes longer, increasing time-to-first-token. Net effect is closer to 4–8× cost in practice.

Estimate GPU memory needed to serve Llama 70B at 4K context, batch size 16, fp16.

Framework: (1) Model weights: 70B × 2 bytes = 140 GB. (2) KV cache per request: 2 × 80 layers × 8 heads × 128 dim × 4096 × 2 bytes ≈ 670 MB. Times 16 = ~11 GB. (3) Activations: small relative to the above. Total ≈ 150+ GB. Fits across 2× H100 (80GB each) with tensor parallelism. Show the math even if numbers are approximate — the method is what matters.

When should you use embeddings vs full LLM generation for a task?

Framework: Embeddings for retrieval, similarity, classification, clustering — anything where you need a vector representation. LLM generation for open-ended text production. RAG combines them: embeddings retrieve relevant chunks, LLM generates with them as context. Cost: embeddings are ~1000× cheaper per query than generation.

Strong answer vs weak answer.

Weak "Transformers use attention to process sequences. They're good at language tasks."
Strong "Transformers stack self-attention + FFN layers. Attention is O(n²) in sequence length, which dominates at long contexts. KV cache makes decode O(n) instead of O(n²) but adds memory cost linear in concurrent requests. The system tradeoffs are model size (GPU floor), KV cache (concurrency ceiling), and context length (squeezes both)."

Mistakes candidates make.

  • Treating transformers as a black box. Interviewers want systems intuition, not math derivation.
  • Not knowing the rough cost of attention (O(n²)) and why it dominates at long context.
  • Forgetting the KV cache exists. It's THE central concept of efficient inference.
  • Confusing model size and KV cache size. Both contribute to GPU memory but scale differently.
  • Not asking about latency vs throughput tradeoffs. Different optimizations help different metrics.

Chapter Capstone
CAPSTONE · END OF CHAPTER I

Design a globally distributed AI API service.

A production-grade system design problem that exercises every concept from Weeks 1–4. Treat it as if a Staff Engineer interview is on the line.

Requirements

  • Serve a 70B-parameter LLM to global users with P95 time-to-first-token under 500ms.
  • 100K concurrent users at peak; 10× burst capacity required during events.
  • Multi-region: US, EU, Asia. Each region must serve traffic locally even if cross-region links fail.
  • Per-user inference logs must be durable, queryable for 90 days, replayable for training data curation.
  • Model versions must be globally consistent — no region serves the wrong version.
  • Hot users (top 0.1% by request rate) should not degrade service for others.

Architecture sketch

Consistency decisions

  • Model registry (PC/EC): Strong consistency mandatory — no region can serve the wrong version. Spanner-like store. Updates take seconds globally; that's fine because deploys aren't frequent.
  • Model artifacts (eventual): Multi-GB weight files replicated to regional CDNs. Pinned by content hash. Async replication, but every region verifies hash before serving.
  • Inference logs (at-least-once + idempotent): Stamped with UUIDs at write time. Cross-region mirror to central data lake for training data curation.
  • KV cache (per-pod, ephemeral): Lost on pod failure. Consistent hashing on user_id keeps cache warm; a hot user override pool prevents one user from overwhelming a single pod.

Scaling strategy

  • Sharding: Inference pods sharded via consistent hashing on user_id with 256 vnodes per pod. Whales detected by per-second request rate get round-robin'd across pods, accepting cache loss.
  • Tensor parallelism: 70B model split across 8 GPUs per pod (TP=8). Tested at p95 TTFT < 400ms for typical 1K-token prompts.
  • Continuous batching: vLLM with paged KV cache, max batch 32, max in-flight tokens 8K per pod.
  • Burst handling: Pod autoscaling on queue depth + GPU utilization. Pre-warmed reserve pool of 20% capacity always idle for instant burst.
  • Hot-key isolation: Top 0.1% users routed to dedicated pod pool; regular users to general pool.

Failure handling

  • Region partition: Each region keeps serving with last-known model version. Inference logs queue locally; mirror catches up when partition heals. Model registry reads use cached version (with TTL); writes blocked until quorum restored.
  • Pod death: Consistent hashing redistributes load to neighbors; KV cache cold-start for affected users (degraded latency, not failure).
  • Kafka lag: Alert at 30 min lag; auto-scale consumers. If retention is at risk, drop debug-level logs first, then sample 10% of routine logs.
  • Cascading retries: Every service call wrapped in circuit breaker. Bounded retries with jittered exponential backoff. 429 returned to clients before queues saturate.
  • Long-prompt OOM: Hard cap at 100K tokens. Reject with explicit error before pod is at risk. Per-request memory budget enforced at admission.

What you should be able to defend

  • Why the model registry is PC/EC (cost is tolerable, correctness is mandatory).
  • Why KV cache stickiness is worth the hot-key risk (cost dominates without it).
  • Why at-least-once + idempotent is preferred over exactly-once (operational simplicity, equivalent correctness).
  • How you'd know the system is healthy at 3 a.m. (lag, utilization, p95 TTFT, error budget).
